**Results:**

| Port     | State | Service      | Version                                    |
|----------|-------|--------------|--------------------------------------------|
| 21/tcp   | open  | FTP          | ProFTPD — `ftp.int.inlanefreight.htb`      |
| 22/tcp   | open  | SSH          | OpenSSH 8.2p1 Ubuntu 4ubuntu0.2            |

**Key observations:**

- The banner on port **21** leaks an internal hostname: `ftp.int.inlanefreight.htb` → useful for DNS enumeration against port 53 (BIND 9.16.1).
- Port **2121** exposes a separate FTP instance named *Ceil's FTP*, likely configured specifically for user `ceil`.
- Port **53** running ISC BIND opens DNS enumeration scenarios (zone transfer, internal records) — a vector to investigate separately.

> 💡 **OPSEC Note:** The scan uses `-sS` (SYN scan) and `-sV` (service detection). On production environments, consider throttling with `--max-rate` to reduce noise.

## 3. FTP Exploration — SSH Private Key Retrieval

Once authenticated, we list all files including hidden ones:

```bash
ftp> ls -la
ftp> cd .ssh
ftp> ls -la
```

<img width="1922" height="742" alt="exploring ftp" src="https://github.com/user-attachments/assets/9f46181b-6ff4-4ea2-90c4-2519699b7320" />


Contents of the `.ssh` directory:

| File              | Permissions  | Notes                         |
|-------------------|--------------|-------------------------------|
| `authorized_keys` | -rw-rw-r--   | Authorized public keys        |
| `id_rsa`          | -rw-------   | ⚠️ RSA private key            |
| `id_rsa.pub`      | -rw-r--r--   | Corresponding public key      |

> ⚠️ **Critical Finding:** The private key `id_rsa` is readable via FTP. This is a severe misconfiguration: an authenticated FTP service with access to the home directory effectively exposes the user's SSH credentials.

**Extracting the key:**

```bash
ftp> get id_rsa
```

```
226 Transfer complete.
3381 bytes received in 00:00 (48.22 KiB/s)
```

## 4. SSH Login via Private Key

Before using the key, we set the correct permissions (SSH rejects keys with overly permissive access):

```bash
chmod 600 id_rsa
ssh -i id_rsa ceil@10.129.33.148
```

## 6. Vulnerability Summary

| # | Finding | Severity | Detail |
|---|---------|----------|--------|
| 1 | FTP exposed with home directory access | 🔴 High | FTP root maps directly to home, exposing `.ssh/` |
| 2 | SSH private key readable via FTP | 🔴 Critical | `id_rsa` downloadable without restrictions → SSH escalation |
| 3 | FTP banner leaks internal hostname | 🟡 Medium | `ftp.int.inlanefreight.htb` exposed in banner → info leak |
| 4 | DNS BIND 9.16.1 exposed on port 53 | 🟡 Medium | Potential zone transfer on `inlanefreight.htb` and `int.inlanefreight.htb` |

---

## 7. Lessons Learned

- An authenticated FTP service does not equal a secure one: if the FTP root coincides with the user's home directory, **everything** is potentially exposed.
- **Service banners** are a primary intelligence source: they reveal hostnames, versions, and internal network structure.
- The presence of BIND on port 53 is a vector not to underestimate — a **DNS Zone Transfer** could expose the entire internal DNS map.
